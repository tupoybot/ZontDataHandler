# Промпт для Codex на этап 1

```text
Ты работаешь в пустом репозитории Android-проекта. Нужно реализовать минимальный, но рабочий MVP приложения, которое получает текущие данные из API ZONT и отдаёт их в Wear OS complications.

Перед началом реализации обязательно прочитай:
- `docs/stage-01-plan.md`
- `docs/development-plan.md`

Сначала согласуй свою реализацию с этими документами, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких секретов в git;
- приложение не нужно публиковать в Play Market;
- пользователь будет тестировать сам на Android emulator, Galaxy S23+ и Galaxy Watch Ultra;
- не нужно делать лишних усложнений;
- нужна именно рабочая вертикаль phone -> watch complications;
- `wear`-приложение нужно и должно участвовать в работе решения, но полноценный UX на часах делать не нужно.

Что нужно сделать:
1. Создай Android-проект на Kotlin с модулями `mobile`, `wear` и `shared`.
2. В `mobile` сделай простой экран настроек/статуса на Compose:
   - `X-ZONT-Client`
   - `X-ZONT-Token`
   - `device_id`
   - `zone` со значением по умолчанию `1`
   - настройка частоты автообновления со значением по умолчанию `5 минут`
   - кнопка `Refresh`
   - отображение последнего snapshot и времени обновления
3. Секреты храни только локально на устройстве. Не хардкодь их в коде, ресурсах, `BuildConfig` или gradle-файлах.
4. Реализуй запрос к ZONT API:
   - `POST https://my.zont.online/api/devices`
   - body: `{"load_io": true}`
   - headers: `X-ZONT-Client`, `X-ZONT-Token`, `Content-Type: application/json`
5. Из ответа собери нормализованный snapshot с полями:
   - `roomTemperature`
   - `burnerModulation`
   - `targetTemperature`
   - `coolantTemperature`
   - `deviceId`
   - `updatedAtEpochSeconds`
   - `isStale`
   - `errorMessage` или аналог
6. Для этапа 1 используй такие правила маппинга:
   - `targetTemperature` <- `io.last-boiler-state.target_temp`
   - `burnerModulation` <- `io.last-boiler-state.ot.rml`
   - `coolantTemperature` <- `io.last-boiler-state.ot.bt`; если нет, fallback на `thermometers[]` с `functions[].f == "heat"` и `zone == 1`
   - `roomTemperature` <- `thermometers[]` с `functions[].f == "control"` и `zone == 1`; если нет, fallback на первый доступный датчик `control`
7. Ошибки и частично пустые данные обрабатывай мягко:
   - приложение не должно падать
   - если конкретная метрика отсутствует 2 обновления подряд, показывай `--`
   - сохраняй описание последней ошибки
8. Добавь автообновление через `WorkManager` с настраиваемым интервалом:
   - по умолчанию `5 минут`
   - ручной `Refresh` должен остаться
   - логика placeholder/stale должна учитывать выбранную частоту обновления
9. Передавай последний успешный snapshot на часы через Wear Data Layer.
10. В `wear` реализуй 5 complication providers:
   - 1 combined provider для большого верхнего слота
   - 4 отдельных providers: room temp, burner modulation, target temp, coolant temp
11. Требования к complications:
   - combined provider должен поддерживать `LONG_TEXT` и при необходимости `SHORT_TEXT`
   - одиночные providers должны поддерживать `SHORT_TEXT`
   - если данные устарели или отсутствуют, это должно быть понятно по тексту
   - обновляй активные complications после получения нового snapshot на watch side
12. Добавь минимальный `README.md`:
   - как собрать проект
   - как локально запустить и проверить проект
   - как заполнить настройки
   - как настроить интервал автообновления
   - как проверить complications вручную
13. Добавь разумный `.gitignore`.

Ограничения:
- не делай сервер, базу данных и сложную многомодульную архитектуру сверх необходимого;
- не добавляй авторизацию по логину/паролю, достаточно токена;
- не делай графики, историю и управление котлом;
- не пытайся публиковать приложение;
- не используй реальные секреты в репозитории.

Технические предпочтения:
- Kotlin
- Jetpack Compose для простого UI
- простой и надёжный HTTP-стек
- сериализуемая shared-модель snapshot
- Wear OS complications API

Что важно по UX:
- combined complication должен быть пригоден для верхнего большого слота Samsung watch face;
- отдельные complications должны быть удобны для круглых слотов;
- значения форматируй коротко и читаемо, например `21.5°`, `42%`, `37°`;
- когда значение скрывается из-за 2 подряд пропусков, показывай короткую заглушку `--`

Что проверить перед завершением:
- проект собирается;
- нет хардкоженных секретов;
- есть понятный путь ручной проверки;
- документируй все сделанные допущения в README или в финальном отчёте.

Если какие-то Android/Wear детали окажутся спорными, выбирай самое простое решение, которое даёт рабочий MVP и не требует лишней инфраструктуры.
```
