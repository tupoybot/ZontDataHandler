# Промпт для Codex на этап 2

```text
Ты работаешь в уже существующем Android-проекте, где этап 1 завершён и рабочая вертикаль `phone -> watch complications` уже реализована.

Перед началом реализации обязательно прочитай:
- `docs/stage-02-plan.md`
- `docs/development-plan.md`
- при необходимости текущую реализацию этапа 1

Сначала согласуй свою реализацию с этими документами, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких секретов в git;
- приложение не нужно публиковать в Play Market;
- этап 1 уже работает, его нельзя ломать;
- пользователь тестирует на Android emulator, Wear OS emulator, Galaxy S23+ и Galaxy Watch Ultra;
- на этапе 2 главный фокус не "добавить ещё фич", а сделать complications более нативными и аккуратными;
- `wear`-приложение остаётся обязательной частью решения.

Что нужно сделать:
1. Сохрани рабочую синхронизацию `phone -> watch` и существующие 5 complication providers.
2. Переработай presentation complications так, чтобы они выглядели более нативно на Wear OS и особенно на Samsung watch face.
3. Убери текстовые подписи из complication output:
   - не показывай `ROOM`, `BURNER`, `TARGET`, `COOLANT` как основной видимый текст;
   - в combined complication не используй строку вида `Room 23.7° | Burner 48% | ...`;
   - максимум допустимы пиктограммы или очень короткая общая метка, если это реально улучшает результат.
4. Для отдельных complications сделай приоритет:
   - пиктограмма;
   - под ней значение;
   - избегай лишних title/subtitle, если без них вид нативнее.
5. Для combined complication:
   - поддержка `LONG_TEXT` обязательна;
   - при необходимости оставь `SHORT_TEXT` fallback;
   - presentation должен быть компактным и ориентированным на большой верхний слот Samsung watch face;
   - старайся показывать только значения, без расшифровывающих слов.
6. Missing/stale presentation оставь понятным, но лаконичным:
   - отсутствующее значение -> `--`;
   - stale должен быть заметен, но без длинных слов.
7. Если нужно, добавь минимальный набор монохромных векторных иконок для watch complications.
8. Не ломай подтверждённый маппинг реального устройства:
   - `roomTemperature` должен остаться комнатной температурой;
   - `burnerModulation` должен остаться модуляцией горелки;
   - `targetTemperature` для подтверждённого `z3k-state` устройства должен оставаться уставочной температурой теплоносителя (`setpoint_temp`), а не целевой температурой помещения;
   - `coolantTemperature` должен оставаться фактической температурой теплоносителя.
9. При необходимости подчисти watch status screen, но без превращения его в полноценный отдельный UI.
10. Обнови `README.md`, если изменятся:
   - способ ручной проверки;
   - визуальная логика complications;
   - рекомендации по тестированию на эмуляторе и Galaxy Watch.
11. Исправь подтверждённый баг текущей реализации:
   - если пользователь ставит интервал автообновления меньше `5` минут, например `1` минуту для теста, после `Refresh` значение не должно самопроизвольно возвращаться на `5`, если только такое ограничение не сделано осознанно и явно не показано пользователю;
   - если оставляешь ограничение, оно должно быть явно оформлено в UI/логике и не выглядеть как случайный сброс.
12. Добавь еще один маленький compication, в котором будет 2 температуры: targetTemperature и coolantTemperature. Одна под другой или как-то ещё там обыграй

Ограничения:
- не переделывай архитектуру заново;
- не добавляй сервер, БД и лишнюю инфраструктуру;
- не возвращай длинные подписи ради "понятности";
- не делай перегруженный декоративный дизайн;
- не используй реальные секреты в репозитории.

Технические предпочтения:
- Kotlin;
- текущая архитектура проекта должна остаться узнаваемой;
- использовать существующий Wear OS complications API проекта;
- минимальные и понятные изменения без переусложнения.

Что важно по UX:
- complications должны выглядеть ближе к нативным Samsung/Wear OS решениям;
- отдельные круглые слоты должны в первую очередь показывать значение, а не подпись;
- combined complication должен выглядеть компактно и спокойно, без длинной расшифровки метрик;
- если нужен текст, он должен быть короче и чище, чем на этапе 1;
- подписи метрик лучше заменить иконкой или убрать совсем.

Что проверить перед завершением:
- проект собирается;
- старая работающая вертикаль не сломана;
- данные по-прежнему доходят до часов;
- combined complication стал визуально чище;
- одиночные complications больше не выглядят как debug-лейблы;
- `targetTemperature` остался корректным для подтверждённого реального устройства;
- баг с откатом интервала после `Refresh` больше не воспроизводится или явно и понятно оформлен как сознательное ограничение;
- путь проверки на Galaxy Watch / Samsung Remote Test Lab описан.

Если какие-то детали API complications спорные, выбирай самое простое решение, которое даёт заметно более нативный результат без новой инфраструктуры.
```
