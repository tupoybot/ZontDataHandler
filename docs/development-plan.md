# ZONT -> Wear OS Complications: общий план разработки

## Цель

Сделать Android-приложение, которое:

- получает текущие данные из API ZONT;
- нормализует их в один компактный набор метрик;
- передаёт эти данные в Wear OS как complications;
- даёт 1 большой комбинированный complication и 4 отдельных круглых complication.

Целевые показатели:

1. температура в помещении;
2. модуляция горелки;
3. температура уставки;
4. температура теплоносителя.

## Что берём из API

Для MVP достаточно опираться на `POST https://my.zont.online/api/devices` с параметром `{"load_io": true}`.

По документации ZONT:

- `io.last-boiler-state.target_temp` -> температура уставки;
- `io.last-boiler-state.ot.rml` -> модуляция горелки OpenTherm;
- `io.last-boiler-state.ot.bt` -> фактическая температура теплоносителя;
- `thermometers[].last_value` -> текущие температурные датчики.

Рабочее предположение для первого релиза:

- температура в помещении: первый активный датчик `thermometers[]`, у которого `functions[].f == "control"` и `zone == 1`;
- температура теплоносителя: сначала `io.last-boiler-state.ot.bt`, если её нет, то датчик `thermometers[]` с `functions[].f == "heat"` и `zone == 1`.

Это предположение сделано по документации ZONT и должно быть подтверждено на вашем реальном устройстве.

## Архитектура без лишних усложнений

Рекомендуемая структура:

- `mobile` -> настройки, запросы к ZONT, хранение последнего снимка данных, синхронизация с часами;
- `wear` -> complication data sources и минимальный экран статуса;
- `shared` -> общие модели данных и константы для связи phone <-> watch.

Технологический минимум:

- Kotlin;
- Gradle Android project;
- Jetpack Compose для простого UI на телефоне и, при необходимости, на часах;
- OkHttp/Retrofit + kotlinx serialization или другой простой HTTP-стек;
- WorkManager для периодического автообновления на телефоне;
- Wear Data Layer для передачи снимка на часы;
- Wear OS Complications API (`SuspendingComplicationDataSourceService` или эквивалентный актуальный API).

## Политика автообновления и заглушек

Для MVP принимаем такие правила:

- автообновление настраивается пользователем в приложении;
- значение по умолчанию: 1 обновление раз в 5 минут;
- если конкретный показатель отсутствует 2 обновления подряд, в UI и в complication показывается заглушка;
- заглушка должна быть короткой и читаемой, например `--`;
- отдельно нужно показывать признак устаревших данных, если весь snapshot давно не обновлялся.

## Предлагаемые этапы

### Этап 1. MVP-вертикаль

Сделать минимальный рабочий путь от ZONT до complication:

- создать проект `mobile + wear + shared`;
- добавить экран настроек на телефоне;
- принимать `X-ZONT-Client`, `X-ZONT-Token`, `device_id`, `zone`;
- дать настройку частоты автообновления, по умолчанию 5 минут;
- забирать текущие данные из `devices(load_io=true)`;
- собрать нормализованный `snapshot` из 4 нужных метрик;
- учитывать политику пропусков и заглушек после 2 подряд отсутствующих значений;
- запускать автообновление на телефоне;
- передавать `snapshot` на часы;
- отдать 1 большой combined complication;
- отдать 4 отдельных complication по одной метрике;
- показать на телефоне простой debug/status screen;
- добавить README с шагами установки и ручной проверки.

### Этап 2. Надёжность и удобство

- ручной refresh;
- отображение ошибок и признака устаревших данных;
- выбор устройства из списка, если у пользователя их несколько;
- понятные подписи и форматирование для watch face;
- fallback-логика при частично отсутствующих полях OpenTherm.

### Этап 3. Полировка и сопровождение

- чистка UI и ресурсов;
- install-инструкции для телефона и часов без Play Market;
- минимальные unit-тесты на парсинг и маппинг snapshot;
- optional: GitHub Actions только на сборку debug/release APK, без секретов.

### Этап 4. Visual polish complications

- обновить проблемные иконки из `assets`;
- поправить визуальную иерархию "текущее значение vs уставка";
- привести combined complication к стабильному двухстрочному виду;
- установить собранные APK на доступные эмуляторы и реальные устройства как обязательную часть проверки.

### Этап 5. Galaxy Ultra и special slots

- подробно исследовать нужный Samsung / Galaxy Watch Ultra slot;
- проверить, можно ли показать там наш provider, или это ограничение stock watch face;
- отдельно развести обычные большие `LONG_TEXT`-слоты и private Samsung slots, чтобы нижний большой slot на другом face не считать "Samsung-only" по умолчанию;
- отдельно отладить поведение `ZONT overview + icons`, если slot/watch face сводит его к нежелательному text-only fallback;
- сохранить рабочую вертикаль и не превращать этот этап в разработку собственного watch face.

## Что не нужно в первом приближении

- публикация в Play Market;
- серверная часть;
- управление котлом из приложения;
- история, графики и аналитика;
- сложная авторизация с логином/паролем, если можно начать с ручного ввода токена;
- сложная локальная БД;
- полноценный пользовательский интерфейс на часах сверх того, что нужно для работы complications и отладки.

## Безопасность и репозиторий

Так как репозиторий будет публичным:

- не хранить реальные токены, логины и пароли в коде, `gradle.properties`, ресурсах или `BuildConfig`;
- любые локальные dev-файлы с секретами класть только в gitignored-пути;
- значения для API хранить на устройстве пользователя локально;
- в README дать пример с заглушками, а не с реальными значениями.

Практичный вариант для MVP:

- секреты вводятся через экран настроек приложения;
- сохраняются только локально на устройстве;
- в репозитории остаются только шаблоны и документация.

## Уточнение по Wear OS части

`wear`-приложение нужно и используется как часть решения.

Для MVP достаточно такого состава:

- watch-side приложение/модуль;
- watch-side компонент с data source providers;
- синхронизация данных с телефона на часы;
- без полноценного пользовательского watch UI, если он не нужен для отладки.

## Критерий готовности проекта

Проект можно считать готовым, когда:

- телефонное приложение получает актуальные данные ZONT для нужного устройства;
- пользователь может настроить частоту автообновления, по умолчанию 5 минут;
- при 2 подряд пропусках конкретного показателя в UI и complications показывается заглушка;
- на часах доступны как минимум 5 complication providers:
  - 1 combined provider для большого верхнего слота;
  - 4 отдельных provider'а для круглых слотов;
- обычные большие слоты watch face, которые открывают стандартный picker и принимают `LONG_TEXT`, считаются рабочим сценарием для наших combined/paired providers;
- `wear`-приложение существует и достаточно для работы providers и синхронизации;
- полноценный пользовательский интерфейс на часах не обязателен;
- в репозитории нет секретов;
- есть краткая инструкция по ручной проверке на эмуляторе, `Galaxy S23+` и `Galaxy Watch Ultra`.
