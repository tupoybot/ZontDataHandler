# Этап 9: план реализации для Codex

## Цель этапа

Этап 9 нужен как опциональное продолжение после complication-focused этапа 8.

Главный смысл этапа 9:

- сделать phone-side настройку заметно удобнее для реального пользователя;
- убрать остатки developer-oriented UX вроде ручного ввода `device_id` там, где приложение уже может помочь само;
- улучшить первый запуск, не трогая подтверждённую рабочую вертикаль `phone -> watch -> complications`;
- добавить ожидаемое для Wear OS поведение: по нажатию на complication открывать приложение на часах.

Этап 9 не должен превращаться в переписывание приложения, серверный проект или новый большой продуктовый релиз. Это bounded usability/onboarding stage.

## Стартовая точка к началу этапа 9

К началу этапа 9 считаем подтверждённым:

- этапы 1-8 завершены;
- рабочая вертикаль `phone -> watch -> complications` уже реализована и проверена;
- login/password flow и ручной `X-ZONT-Token` fallback уже есть;
- приложение уже умеет загрузить `devices[]` после успешного `Get token`;
- complication provider set после stage 8 уже стабилизирован:
  - `ZONT overview`
  - `ZONT color overview`
  - `ZONT room`
  - `ZONT burner`
  - `ZONT setpoint`
  - `ZONT coolant`
  - `ZONT setpoint + coolant`
  - `ZONT room + air setpoint`;
- visual baseline stage 8 уже считается зафиксированным и не должен переоткрываться без прямой причины, кроме wiring tap action;
- release path и documentation baseline уже существуют;
- базовые risks этапов 1-8 уже не должны снова открываться без необходимости.

## Главные вопросы этапа 9

### 1. Выбор устройства вместо ручного `device_id`

Нужно проверить:

- можно ли после `Get token` стабильно показывать пользователю список доступных устройств;
- можно ли выбирать активное устройство из UI, а не только через ручной ввод `device_id`;
- как вести себя, если устройств несколько, список пуст или запрос к `devices[]` временно не удался.

### 2. Первый запуск и onboarding

Нужно проверить:

- какие шаги нового пользователя сейчас выглядят "техническими";
- что можно упростить без скрытой магии;
- как объяснить разницу между login/password flow, ручным токеном и обязательным `Refresh`.

### 3. Диагностика и понятный статус

Нужно проверить:

- хватает ли текущего статуса на телефоне, чтобы понять, что именно сломалось;
- можно ли сделать sync/status screen более дружелюбным без перегрузки;
- где полезно показывать "последний успешный refresh", "последний sync на часы" и активное устройство.

### 4. Открытие приложения на часах по нажатию на complication

Нужно проверить:

- какой существующий watch-side экран логичнее использовать как tap target;
- можно ли добавить это поведение для текущих complications без ломки providers и без нового полноценного watch UI;
- не создаёт ли конкретный watch face или тип slot особых ограничений для tap action.

## Принятые ограничения

Чтобы этап 9 не расползся:

- не менять подтверждённый смысл метрик;
- не переписывать архитектуру `mobile + wear + shared`;
- не добавлять сервер, backend или облачную прослойку;
- не делать новый полноценный watch UI;
- не переоткрывать visual-polish задачи stage 8 без прямой необходимости для tap action;
- не превращать onboarding в многоэкранный продуктовый wizard без ясной пользы;
- не делать commit, tag, merge или push без прямой команды пользователя.

## Scope этапа 9

### 1. Добавить UI-выбор устройства

Нужно:

- использовать уже доступный результат `devices[]`;
- дать пользователю выбрать активное устройство из списка;
- сохранить ручной ввод `device_id` как fallback только там, где автоматический путь реально не сработал.

### 2. Подчистить первый запуск

Нужно:

- упростить последовательность действий после установки;
- сделать сообщения после `Get token` и `Refresh` менее техническими;
- не заставлять пользователя догадываться, какой следующий шаг нужен.

### 3. Улучшить phone-side status

Нужно:

- явно показывать выбранное устройство;
- яснее показывать причину ошибок refresh/sync;
- при возможности разделить "данные обновились на телефоне" и "snapshot ушёл на часы".

### 4. Добавить tap action для complications

Нужно:

- по нажатию на complication открывать существующее приложение/экран на часах;
- придерживаться стандартного Wear OS ожидания, не изобретая отдельный watch-only сценарий;
- не ломать текущее complication behavior и не требовать для этого нового большого watch UI.

### 5. Обновить documentation и manual check

Если UX меняется, нужно синхронно обновить:

- README;
- manual check;
- development plan;
- при необходимости screenshots phone-side интерфейса.

## Критерии приёмки этапа 9

Этап 9 можно считать успешным, если:

- проект по-прежнему собирается;
- рабочая вертикаль `phone -> watch` не сломана;
- пользователь может выбрать устройство понятнее, чем через ручной `device_id` в happy-path сценарии;
- первый запуск стал проще и понятнее;
- статус и ошибки на телефоне стали менее "техническими" и более полезными;
- нажатие на complication открывает приложение на часах в поддерживаемом пользовательском сценарии;
- ручной fallback для сложных случаев не потерян;
- documentation обновлена согласованно.

## Как тестировать на этапе 9

Минимальная обязательная проверка:

- пройти сценарий "новая установка -> Get token -> выбор устройства -> Refresh";
- проверить сценарий с одним устройством;
- проверить сценарий с несколькими устройствами;
- проверить fallback, если `devices[]` не удалось загрузить;
- убедиться, что snapshot по-прежнему доходит до часов;
- проверить, что нажатие на complication открывает приложение на часах.

## Что не нужно делать на этапе 9

- не строить новый account-management модуль;
- не делать сложный multi-user режим;
- не превращать settings screen в перегруженную админку;
- не открывать снова задачи этапов 5-8 без явной причины.
