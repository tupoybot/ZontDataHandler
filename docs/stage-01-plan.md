# Этап 1: план реализации для Codex

## Цель этапа

Сделать простой, но рабочий MVP, который уже можно поставить на телефон и часы и вручную проверить:

- телефон получает данные из ZONT;
- телефон хранит последний нормализованный snapshot;
- snapshot уходит на часы;
- часы отдают combined complication и 4 отдельных complication.

## Принятые ограничения

Чтобы не раздувать объём первого этапа:

- поддерживаем только токеновую авторизацию `X-ZONT-Token`;
- не делаем получение токена по логину/паролю;
- не делаем управление котлом;
- не делаем историю и графики;
- считаем основной зоной `zone = 1`;
- поддерживаем одного активного пользователя и одно выбранное устройство;
- `wear`-приложение делаем обязательно, но без полноценного пользовательского watch UI сверх необходимого для complications.

## Scope этапа

### 1. Создать каркас Android-проекта

Нужно создать минимальный проект с модулями:

- `mobile`;
- `wear`;
- `shared`.

## 2. Описать общую модель snapshot

В `shared` нужен компактный сериализуемый объект, например:

- `roomTemperature`;
- `burnerModulation`;
- `targetTemperature`;
- `coolantTemperature`;
- `deviceId`;
- `updatedAtEpochSeconds`;
- `isStale`;
- `sourceSummary` или `errors`.

## 3. Реализовать настройки на телефоне

Нужен простой экран настроек/статуса, где можно:

- ввести `X-ZONT-Client`;
- ввести `X-ZONT-Token`;
- указать `device_id`;
- указать `zone`, по умолчанию `1`;
- настроить частоту автообновления, по умолчанию `5 минут`;
- нажать `Refresh`;
- увидеть последние полученные значения и время обновления.

Секреты не должны попадать в git. Хранить их только локально на устройстве.

## 4. Реализовать запрос к ZONT API

Минимально нужен вызов:

- `POST https://my.zont.online/api/devices`
- body: `{"load_io": true}`
- headers:
  - `X-ZONT-Client`
  - `X-ZONT-Token`
  - `Content-Type: application/json`

Нужна только та часть парсинга, которая требуется для 4 метрик.

## 5. Нормализовать нужные показатели

Правила маппинга для этапа 1:

- `targetTemperature` <- `io.last-boiler-state.target_temp`;
- `burnerModulation` <- `io.last-boiler-state.ot.rml`;
- `coolantTemperature` <- `io.last-boiler-state.ot.bt`, fallback на датчик `thermometers[]` с функцией `heat` и `zone == 1`;
- `roomTemperature` <- датчик `thermometers[]` с функцией `control` и `zone == 1`, fallback на первый доступный `control`.

Если какое-то поле отсутствует:

- не падать;
- возвращать `null`;
- явно помечать snapshot как частично заполненный или stale/error.

Правило заглушек для этапа 1:

- если конкретная метрика отсутствует 2 обновления подряд, в UI и в complication показывать заглушку `--`;
- если позже значение снова приходит, заглушка заменяется реальным значением;
- дефолтный порог в MVP считается от выбранной частоты автообновления, по умолчанию это 10 минут до перехода в заглушку.

## 6. Передать snapshot на часы

Нужен простой канал phone -> watch:

- после успешного refresh на телефоне отправить snapshot в Wear Data Layer;
- на часах хранить последнюю полученную копию snapshot;
- при получении новых данных обновлять активные complications.

Автообновление для этапа 1 обязательно:

- запускать через `WorkManager`;
- использовать настройку интервала из приложения;
- значение по умолчанию `5 минут`.

## 7. Реализовать Wear OS complications

Нужно сделать 5 providers:

- 1 combined provider для большого слота;
- 1 provider "Room temperature";
- 1 provider "Burner modulation";
- 1 provider "Target temperature";
- 1 provider "Coolant temperature".

Минимальные требования:

- combined provider должен уметь отдать `LONG_TEXT` и, если нужно, `SHORT_TEXT`;
- одиночные providers должны уметь отдать `SHORT_TEXT`;
- для отсутствующих данных показывать `--`;
- для устаревших данных показывать понятный признак stale.

## 8. Добавить минимальные экраны статуса

На телефоне:

- настройки;
- текущий snapshot;
- ошибка последнего обновления, если она есть.

На часах:

- можно ограничиться очень простым экраном-заглушкой или экраном статуса;
- `wear`-приложение обязательно;
- полноценный watch UI не нужен.

## 9. Документация и git hygiene

Нужно добавить:

- `.gitignore` для локальных секретов и Android build-артефактов;
- `README.md` с инструкцией:
  - как собрать;
  - как установить и запустить проект для локальной проверки;
  - как заполнить настройки;
  - как настроить интервал автообновления;
  - как добавить complications на watch face;
  - как вручную проверить результат.

## Критерии приёмки этапа

Этап считается завершённым, если:

- проект собирается локально;
- телефон может получить ответ от `devices(load_io=true)`;
- на телефоне видны 4 целевые метрики и timestamp последнего обновления;
- на телефоне есть настройка автообновления с дефолтом `5 минут`;
- автообновление реально выполняется;
- если метрика отсутствует 2 обновления подряд, UI и complications показывают `--`;
- snapshot передаётся на часы;
- на часах доступны 5 complication providers;
- `wear`-приложение/модуль существует и участвует в синхронизации данных;
- combined complication можно поставить в большой слот;
- отдельные complication можно поставить в круглые слоты;
- при отсутствии данных UI и complications не падают;
- в репозитории нет реальных секретов.

## Что можно отложить на потом

- выбор устройства из списка вместо ручного `device_id`;
- получение токена по логину/паролю;
- unit-тесты поверх всей архитектуры.

Если останется время в рамках этапа 1, лучший следующий бонус:

- добавить более гибкую политику stale/placeholder отдельно для каждой метрики и для всего snapshot.
