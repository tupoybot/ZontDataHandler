# Промпт для Codex на этап 5

```text
Ты работаешь в уже существующем Android-проекте, где этапы 1, 2, 3 и 4 уже завершены: рабочая вертикаль `phone -> watch -> complications` реализована, визуальные правки этапа 4 уже внесены, а релиз `0.2` подготовлен как текущая базовая точка.

Перед началом реализации обязательно прочитай:
- `docs/stage-05-plan.md`
- `docs/development-plan.md`
- `docs/galaxy-specific.md`
- `docs/stage-04-plan.md`
- `docs/galaxy-complication.png`
- при необходимости текущую реализацию providers в `wear`

Сначала согласуй свою реализацию с этими документами, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких секретов в git;
- приложение не нужно публиковать в Play Market;
- этапы 1-4 уже работают, их нельзя ломать;
- пользователь тестирует на Android emulator, Wear OS emulator, Galaxy S23+ и Galaxy Watch Ultra;
- `wear`-приложение остаётся обязательной частью решения;
- на этапе 5 главный фокус: подробное исследование Galaxy Ultra-specific slot и попытка поместить туда нужный complication/provider;
- в этом же этапе нужно разобраться с `ZONT overview + icons`, потому что пользователю нравится этот вариант, но на части слотов он сейчас показывает не то, что ожидается.

Что уже должно остаться рабочим:
- синхронизация `phone -> watch`;
- текущие providers:
  - `ZONT overview`
  - `ZONT overview + icons`
  - `ZONT room`
  - `ZONT burner`
  - `ZONT setpoint`
  - `ZONT coolant`
  - `ZONT setpoint + coolant`
  - `ZONT room + air setpoint`
- текущая логика placeholder/stale;
- подтверждённый маппинг реального устройства:
  - `roomTemperature` — комнатная температура;
  - `burnerModulation` — модуляция горелки;
  - `targetTemperature` для подтверждённого `z3k-state` устройства — уставка теплоносителя (`setpoint_temp`), а не температура воздуха;
  - `coolantTemperature` — фактическая температура теплоносителя;
  - `roomSetpointTemperature` — отдельная желаемая температура воздуха в помещении.

Что нужно сделать:
1. Сохрани текущую рабочую вертикаль и не ломай существующие providers.
2. Подробно исследуй нужный Samsung / Galaxy Watch Ultra slot из `docs/galaxy-complication.png`:
   - это обычный complication slot или встроенный элемент watch face;
   - какие `ComplicationType` он реально принимает;
   - можно ли туда вывести наш provider.
3. Попробуй минимальными и понятными изменениями сделать так, чтобы нужный provider появился в нужном месте.
4. Если для этого нужны bounded-эксперименты с supported types или отдельным экспериментальным provider'ом, это допустимо, но:
   - не ломай текущие рабочие providers;
   - не превращай этап 5 в новую архитектуру.
5. Отладь `ZONT overview + icons`:
   - выясни, почему на части слотов появляются буквы или text-only fallback вместо ожидаемых иконок;
   - добейся наиболее честного и аккуратного поведения;
   - если конкретный слот принципиально не умеет нужный visual режим, зафиксируй это явно и предложи минимальный fallback.
6. Если правки затрагивают preview data, formatter, manifest, provider types или ресурсы, обнови их согласованно.
7. После каждой успешной сборки устанавливай всё собранное, что подходит по платформе, на доступные target'ы:
   - `mobile` на Android emulator и/или Galaxy S23+;
   - `wear` на Wear OS emulator и/или Galaxy Watch Ultra;
   - если доступны и эмуляторы, и реальные устройства, используй их все.
8. Если debug/release APK конфликтуют между собой по подписи или установке, не скрывай это:
   - зафиксируй ограничение;
   - установи хотя бы тот вариант, который реально используется для ручной проверки.
9. Если после bounded-исследования выяснится, что нужный Samsung slot закрыт для сторонних providers, не маскируй это:
   - зафиксируй точное ограничение;
   - оставь проект в рабочем состоянии;
   - не пытайся изобретать неподтверждённый proprietary Samsung workaround.
10. Не делай commit, tag, merge или push в `main` без явной команды пользователя.

Ограничения:
- не переделывай архитектуру заново;
- не добавляй сервер, базу данных и лишнюю инфраструктуру;
- не ломай текущие complication layouts сверх того, что нужно для Samsung-исследования и исправления `overview + icons`;
- не используй реальные секреты в репозитории;
- не превращай этап 5 в бесконечную борьбу с одним watch face;
- не подменяй результат исследования догадками без практической проверки.

Технические предпочтения:
- Kotlin;
- текущая архитектура проекта должна остаться узнаваемой;
- использовать существующий Wear OS complications API проекта;
- минимальные и понятные изменения без переусложнения;
- если нужен экспериментальный provider, он должен быть явно изолирован и не ломать основной UX.

Что важно по UX:
- текущий рабочий пользовательский путь не должен стать хуже;
- если нужный Samsung slot реально поддерживает наш provider, пользователь должен суметь выбрать и увидеть его там;
- если не поддерживает, это должно быть объяснено честно и конкретно;
- `ZONT overview + icons` должен выглядеть как осмысленный visual variant, а не как буквенная имитация иконок;
- на совместимых слотах нужно предпочесть реальные иконки и аккуратную читаемость, а не случайный fallback.

Что проверить перед завершением:
- проект собирается;
- старая рабочая вертикаль не сломана;
- данные по-прежнему доходят до часов;
- существующие providers работают;
- подтверждённый маппинг метрик не регрессировал;
- исследование нужного Galaxy Ultra slot выполнено на практике, а не только "по коду";
- есть понятный вывод: получилось встроить provider в нужное место или нет;
- `ZONT overview + icons` больше не показывает confusing буквенную легенду там, где можно показать реальную визуальную версию;
- собранные APK установлены на доступные target'ы;
- commit/push не делался без команды пользователя.

Если какие-то детали Samsung slot behavior, provider types или visual fallback спорные, выбирай самое простое решение, которое:
- даёт практическую проверку на реальном устройстве;
- не ломает существующую систему;
- оставляет после себя понятный вывод, а не бесконечное "ещё чуть-чуть покрутить".
```
