# Промпт Для Codex На Этап 9

```text
Ты работаешь в уже существующем Android-проекте, где этапы 1-8 уже завершены, а базовая вертикаль `phone -> watch -> complications` считается рабочей и не должна быть сломана.

Перед началом реализации обязательно прочитай:
- `docs/codex-stage-context.md`
- `docs/stage-09-plan.md`
- `README.md`
- `mobile`-код вокруг settings/status UI, auth/token flow, `devices[]`, refresh и сохранения настроек
- `wear`-код вокруг complications и launch/tap behavior

Важно: единый `docs/codex-stage-context.md` здесь специально заменяет чтение всех предыдущих stage-планов подряд. Не трать время на последовательное чтение `stage-01 .. stage-08`, если только этого отдельно не требует задача.

Сначала согласуй реализацию с этим контекстом, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких secrets в git;
- `wear`-приложение остаётся обязательной частью решения;
- build number уже инкрементируется автоматически между сборками, это поведение нельзя регрессировать;
- signed/unsigned release path уже оформлен и тоже не должен ломаться;
- complication provider set после stage 8 уже стабилизирован и не должен случайно переоткрываться visual-правками.

Что уже должно остаться рабочим:
- синхронизация `phone -> watch`;
- placeholder/stale логика;
- подтверждённый маппинг метрик;
- текущие providers:
  - `ZONT overview`
  - `ZONT color overview`
  - `ZONT room`
  - `ZONT burner`
  - `ZONT setpoint`
  - `ZONT coolant`
  - `ZONT setpoint + coolant`
  - `ZONT room + air setpoint`
- текущая release/debug инфраструктура;
- login/password flow и ручной `X-ZONT-Token` fallback;
- автоподстановка `device_id` после успешного `Get token`, если это возможно.

Главный фокус этапа 9:
1. Сделать phone-side настройку заметно удобнее и менее "технической".
2. Упростить happy path выбора устройства через уже существующий `devices[]`.
3. Сделать первый запуск и status/debug screen понятнее для реального пользователя.
4. Добавить ожидаемое Wear OS поведение: по нажатию на complication открывать приложение на часах.

Что нужно сделать:
1. Сначала проверь, как сейчас реально устроены:
   - mobile settings/status UI;
   - flow `Get token -> devices[] -> device_id`;
   - refresh/status messaging;
   - watch-side launchable activity и complication tap behavior.
2. Не начинай с дизайна "в теории":
   - review-first;
   - затем минимальная bounded реализация;
   - затем emulator/device verification.
3. Для выбора устройства используй уже имеющиеся данные `devices[]`, а не придумывай новый серверный или sync path.
4. Не убирай ручной `device_id` fallback, если автоматический путь временно не сработал.
5. Если устройств несколько, UX должен явно показывать, какое устройство выбрано сейчас.
6. Если `devices[]` пуст или загрузка не удалась:
   - не маскируй это;
   - сохрани понятный fallback;
   - оставь пользователю рабочий путь продолжить настройку вручную.
7. Для phone-side статуса старайся сделать различимыми хотя бы:
   - успешный refresh на телефоне;
   - успешную отправку snapshot на часы;
   - текущее выбранное устройство;
   - понятную причину ошибки, если что-то сломалось.
8. Для tap action используй существующий watch-side экран/входную точку, а не строй новый большой watch UI без необходимости.
9. Если tap action зависит от ограничений конкретного complication type или renderer:
   - проверь это честно;
   - не обещай поведение, которое не подтверждено кодом и проверкой.
10. Если изменение затрагивает README, stage docs, development plan или manual check, обнови их без противоречий.
11. После значимых правок собирай и проверяй:
   - тесты;
   - `mobile`;
   - `wear`;
   - сценарий `Get token -> выбор устройства -> Refresh`;
   - complication tap behavior настолько, насколько это реально проверить локально.
12. По завершении этапа по умолчанию раскатывай только `debug` на эмуляторы.
13. `release` на физические устройства не накатывай без отдельной команды пользователя.
14. Если нужен локальный `adb`, используй уже сохранённый путь/alias из `.codex` memory, а не ищи SDK заново.
15. Не делай commit, tag, merge или push без явной команды пользователя.

Ограничения:
- не добавляй новые продуктовые фичи вне phone-side UX и bounded complication tap behavior;
- не переписывай архитектуру `mobile + wear + shared`;
- не меняй смысл существующих метрик;
- не переоткрывай visual redesign complications без прямой причины;
- не обещай platform behavior, который не подтверждён кодом или проверкой.

Что проверить перед завершением:
- проект собирается;
- тесты проходят;
- старая рабочая вертикаль не сломана;
- новый UX реально упрощает happy path выбора устройства;
- ручной fallback не потерян;
- нажатие на complication либо открывает приложение на часах, либо platform limitation описан честно;
- documentation обновлена согласованно;
- debug раскатан на эмуляторы;
- commit/push не делался без команды пользователя.
```
