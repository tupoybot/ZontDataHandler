# Этап 3: план реализации для Codex

## Цель этапа

Подготовить проект к аккуратному внутреннему релизу `0.1`, не ломая уже рабочую вертикаль `phone -> watch -> complications`.

Главный смысл этапа 3:

- упростить первый запуск приложения для реального пользователя;
- подготовить воспроизводимую сборку и выпуск APK без Play Market;
- привести документацию и release-процесс в порядок;
- отдельно и ограниченно проверить, можно ли поддержать специальный Samsung slot без большой отдельной ветки разработки.

Этап 3 не должен превращаться в новую архитектуру или в бесконечное R&D по watch face-специфике Samsung.

## Что уже считаем завершённым

К началу этапа 3 уже подтверждено:

- проект собирается;
- `mobile`, `wear`, `shared` работают как единая связка;
- синхронизация `phone -> watch` работает;
- на часах уже есть 7 providers:
  - `ZONT overview`;
  - `ZONT room`;
  - `ZONT burner`;
  - `ZONT setpoint`;
  - `ZONT coolant`;
  - `ZONT setpoint + coolant`;
  - `ZONT room + air setpoint`;
- complications presentation на этапе 2 уже был приведён к более нативному виду;
- подтверждённый маппинг реального устройства нельзя ломать:
  - `roomTemperature` остаётся комнатной температурой;
  - `burnerModulation` остаётся модуляцией горелки;
  - `targetTemperature` для подтверждённого `z3k-state` устройства остаётся уставкой теплоносителя (`setpoint_temp`);
  - `coolantTemperature` остаётся фактической температурой теплоносителя;
  - `roomSetpointTemperature` остаётся отдельной желаемой температурой воздуха в помещении.

Этап 3 должен использовать эту базу, а не пересобирать проект заново.

## Принятое решение по Samsung-ветке

В заметках пользователя есть желание попробовать поддержать "особое" место для complication на `Galaxy Watch Ultra` и Samsung Ultra-style watch faces. Это важная задача, но у неё высокий риск расползания scope:

- часть специальных Samsung slots может быть недоступна для сторонних providers вообще;
- для проверки может понадобиться отдельная поддержка `RANGED_VALUE` / `GOAL_PROGRESS`;
- результат может зависеть не от нашего provider, а от самого stock watch face.

Поэтому для этапа 3 принимаем такое решение:

- в этап 3 входит только ограничённое исследование и самый простой технический эксперимент по `docs/galaxy-specific.md`;
- если быстро находится небольшой и безопасный путь, который не ломает текущие providers, его можно включить в этап 3;
- если задача упирается в ограничения stock Samsung face, отдельный специфичный provider-режим или крупную переработку complication types, это нужно явно вынести в этап 4;
- выпуск `0.1` не должен зависеть от того, удастся ли встроиться именно в специальный Samsung slot.

Иными словами: этап 3 делает bounded audit, а не обещает любой ценой "завести Ultra-дугу".

## Принятые ограничения

Чтобы этап 3 остался компактным и полезным:

- не переделывать архитектуру;
- не добавлять сервер, прокси, облако и другую внешнюю инфраструктуру;
- не добавлять секреты в git;
- не публиковать приложение в Play Market;
- не делать полноценный отдельный watch UI;
- не превращать GitHub Actions в сложную release-платформу;
- не тратить весь этап на Samsung-specific ветку, если она быстро не подтверждается.

## Scope этапа

### 1. Упростить получение токена из ZONT

Сейчас рабочий путь требует ручного ввода токена. На этапе 3 нужно попробовать сделать более удобный способ входа:

- добавить в `mobile` возможность получить рабочий токен по логину и паролю ZONT;
- по возможности сохранить существующий ручной ввод токена как fallback;
- не хардкодить логины, пароли, токены и client-id в репозитории;
- пароль не должен попадать в git, логи или документацию;
- если для входа достаточно стандартного HTTP-вызова без новой инфраструктуры, реализовать его прямо в приложении;
- если реальный auth-flow окажется нестабильным, скрытым, завязанным на web-only механику, captcha или другие нежелательные обходы, не городить костыли: оставить ручной fallback и явно задокументировать ограничение.

Практический приоритет:

- сначала сохранить уже рабочий token-based режим;
- затем поверх него добавить более удобный login/password flow, если он реализуем без переусложнения.

### 2. Сохранить и перепроверить текущую рабочую вертикаль

Этап 3 не должен сломать:

- получение данных с телефона;
- передачу snapshot на часы;
- существующие 7 complication providers;
- текущую визуальную логику complications;
- подтверждённый маппинг метрик реального устройства.

Если в рамках auth или Samsung-ветки придётся менять shared-модель, transport или providers, нужно делать это минимально и с регрессионной проверкой.

### 3. Добавить GitHub Actions для APK-сборок

Нужно настроить минимальный и понятный CI:

- сборка `mobile` и `wear`;
- как минимум `debug` APK;
- по возможности также `release` APK для sideload-распространения;
- публикация APK как GitHub Actions artifacts;
- запуск без реальных секретов.

Важно:

- так как приложение не идёт в Play Market, допустимо выбрать максимально простой путь, пригодный именно для локальной установки и GitHub release;
- если "настоящая" release-signing схема без секретов получается чрезмерно неудобной, нужно честно выбрать и задокументировать упрощённую стратегию, пригодную для side-load сценария.

### 4. Почистить и актуализировать README

README нужно привести к состоянию, пригодному для внешнего чтения:

- убрать устаревшие или лишние детали;
- оставить понятный путь сборки, установки и проверки;
- описать текущие providers и реальную логику complications;
- описать путь тестирования на эмуляторах, `Galaxy S23+`, `Galaxy Watch Ultra` и при необходимости Samsung Remote Test Lab;
- добавить честную пометку, что кодовая база полностью сгенерирована с помощью Codex;
- не превращать README в длинный дневник разработки.

### 5. Подготовить релиз `0.1`

Нужно довести проект до состояния "можно выпустить":

- проверить и при необходимости обновить `versionName` / `versionCode`;
- подготовить краткие release notes;
- оформить понятный changelog уровня `0.1`, хотя бы в кратком виде;
- сделать чистый git commit по итогам этапа;
- если локальная среда и права позволяют, создать tag/release `0.1`;
- если автоматический выпуск GitHub release недоступен из текущей среды, подготовить всё необходимое для ручного выпуска без недосказанности.

### 6. Ограниченно проверить Samsung special slot compatibility

Использовать `docs/galaxy-specific.md` как guide для небольшого исследования.

Что входит в этап 3:

- проверить, какие `ComplicationType` уже публикуют существующие providers;
- проверить, есть ли смысл в простой поддержке `RANGED_VALUE` и/или `GOAL_PROGRESS` хотя бы для одного-двух подходящих providers;
- проверить `getPreviewData(...)`, manifest и supported types;
- попробовать минимальный эксперимент, если он не ломает текущие текстовые/круглые слоты.

Что не должно происходить на этапе 3:

- большой отдельный Samsung-only redesign;
- переписывание всех providers под новый набор типов;
- разработка собственного watch face;
- попытка обойти системные ограничения stock Samsung face.

Условие остановки:

- если после короткого и честного исследования видно, что проблема в ограничении конкретного stock watch face или задача требует отдельного большого направления, это фиксируется как stage 4 work item и не блокирует релиз `0.1`.

## Критерии приёмки этапа

Этап 3 считается завершённым, если:

- проект по-прежнему собирается;
- текущая вертикаль `phone -> watch` не сломана;
- существующие 7 providers продолжают работать;
- подтверждённый маппинг метрик не регрессировал;
- для пользователя появился более удобный путь получения токена, либо ограничение auth-flow честно задокументировано, а ручной токеновый путь сохранён;
- появился GitHub Actions workflow для сборки и публикации APK artifacts без секретов;
- README стал короче, чище и актуальнее;
- в README или release notes явно сказано, что код был полностью сгенерирован Codex;
- подготовлены версия `0.1` и материалы для релиза;
- Samsung-ветка либо:
  - дала небольшой безопасный технический результат в рамках этапа 3;
  - либо явно зафиксирована как отдельная задача этапа 4 без размытия текущего релиза.

## Как тестировать на этапе 3

Минимальная обязательная проверка:

- собрать `mobile` и `wear`;
- проверить, что старый token-flow по-прежнему рабочий;
- проверить новый login/password flow, если он реализован;
- установить APK на Android emulator, Wear OS emulator и реальный `Galaxy Watch`, если он доступен;
- убедиться, что после refresh данные по-прежнему доходят до часов;
- убедиться, что выбранные providers видны и обновляются как раньше;
- отдельно проверить CI artifacts и шаги релизной сборки.

Проверка Samsung-ветки:

- если делается эксперимент с новыми complication types, проверить его отдельно и не путать с базовой регрессией текущих providers;
- если результат отрицательный, важно зафиксировать именно вывод, а не оставлять задачу в подвешенном состоянии.

## Что можно вынести в этап 4

- полная отдельная проработка special Samsung slots;
- watch-face-specific режимы отображения;
- расширенная поддержка `RANGED_VALUE` / `GOAL_PROGRESS` для нескольких провайдеров сразу;
- большой CI/release pipeline с полноценной signing-инфраструктурой;
- более глубокие unit/integration tests;
- дополнительная polish-работа по watch UI сверх уже существующего статуса.

## Рекомендуемый приоритет внутри этапа 3

Порядок выполнения лучше держать таким:

1. сохранить текущую рабочую вертикаль и сделать auth-улучшение;
2. настроить GitHub Actions и привести README в порядок;
3. подготовить релиз `0.1`;
4. в оставшийся бюджет времени сделать bounded Samsung audit;
5. если Samsung-ветка быстро не подтверждается, не растягивать этап 3 и сразу формализовать этап 4.
