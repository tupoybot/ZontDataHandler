# Wear OS: кастомные complications для специальных слотов Samsung Watch Ultra / Ultra-style watch faces

## Проблема

Приложение публикует custom complications, но они не отображаются в некоторых специальных слотах циферблата Samsung Watch Ultra / Ultra-style watch faces, например в верхнем секторе-дуге. В системном picker'е для такого слота нельзя выбрать сторонний источник данных.
Нужное дополнение лежит в galaxy-complication.png

Подробный bounded-план следующей итерации теперь вынесен в `docs/stage-05-plan.md`.

## Ключевой вывод

Для таких слотов обычно не нужен отдельный "формат Samsung". Проблема, как правило, в одном из двух вариантов:

1. Слот принимает только ограниченный набор `ComplicationType`, и наш provider не публикует совместимый тип.
2. Слот вообще не является обычным пользовательским complication slot и зарезервирован самим watch face.

## Как это устроено в Wear OS

### 1. Provider публикует только данные и их тип

Приложение-провайдер не управляет тем, где и как именно будет нарисован complication.  
Watch face сам определяет:

- какие слоты существуют
- какие типы данных они поддерживают
- какие слоты доступны пользователю для выбора внешнего provider'а

Следствие: даже корректно реализованный provider может не появляться в picker'е для конкретного слота.

### 2. Provider показывается только при совпадении типов

Системный picker показывает источник только тогда, когда есть пересечение между:

- типами, которые поддерживает слот watch face
- типами, которые публикует наш complication data source

Пример:

- если provider отдаёт только `SHORT_TEXT`
- а слот принимает только `RANGED_VALUE` или `GOAL_PROGRESS`

то provider для этого слота виден не будет.

### 3. Специальные дуговые/краевые слоты могут быть отдельным классом слотов

Верхние дуги, шкалы и edge-like элементы на некоторых циферблатах часто работают как arc/edge complication slots либо как встроенные элементы самого циферблата.

Следствие:

- если слот открыт для внешних providers, нужно поддержать правильные типы данных
- если слот не открыт, принудительно встроиться в него нельзя

## Практическая гипотеза для нашего кейса

Для "веерного" или дугового верхнего слота наиболее вероятны такие ожидания от данных:

- `RANGED_VALUE`
- иногда `GOAL_PROGRESS`

Если provider сейчас публикует только текстовые типы вроде `SHORT_TEXT`, он не появится в picker'е для такого места.

## Что проверить в проекте

### Проверить поддерживаемые типы данных

Нужно убедиться, что data source публикует типы, подходящие для gauge/progress/arc-like слотов.

Минимальный список для проверки:

- `ComplicationType.RANGED_VALUE`
- при необходимости `ComplicationType.GOAL_PROGRESS`
- дополнительно `ComplicationType.SHORT_TEXT` для обычных компактных слотов

### Проверить реализацию data source

Нужно проверить:

- используется корректный `ComplicationDataSourceService` / `SuspendingComplicationDataSourceService`
- корректно реализован `onComplicationRequest(...)`
- реализован `getPreviewData(type)` для всех заявленных типов

### Проверить manifest

Нужно убедиться, что в `AndroidManifest.xml`:

- сервис объявлен как complication data source
- указан permission `com.google.android.wearable.permission.BIND_COMPLICATION_PROVIDER`
- перечислены корректные supported types
- указан icon/preview, если требуется для picker'а

## Ограничения

### Практически подтверждённое ограничение для `Ultra Analog`

На реальном `Galaxy Watch Ultra` текущий stock face `com.samsung.android.watch.watchface.ultraanalog` использует собственную Samsung-конфигурацию слотов.

В on-device asset-конфиге этого face (`UltraAnalogWatchFaceService_settings.xml` внутри APK с часов) для:

- `complication-arc`;
- `complication-left`;
- `complication-right`;
- `complication-bottom`;
- edge-слотов

опубликован жёстко заданный список `Selection` с `category_type=private` и заранее указанными `component_name` для Samsung/system providers.

Практический вывод:

- нужная верхняя дуга `complication-arc` существует как настраиваемый элемент face;
- но для `Ultra Analog` она не ведёт в обычный системный picker сторонних providers;
- сторонний provider вроде `com.botkin.zontdatahandler` туда штатно встроить нельзя без неподтверждённого proprietary workaround, который в проект добавлять не нужно.

### Обычный большой нижний слот на другом watch face

Если после смены циферблата внизу появляется большой слот и он открывает обычный системный picker, это уже не выглядит как special Samsung-only формат.

Практический вывод для проекта:

- такой слот можно использовать для наших задач как обычный `LONG_TEXT` complication slot;
- в текущем приложении для него подходят как минимум `ZONT overview`, `ZONT setpoint + coolant` и `ZONT room + air setpoint`;
- если picker показывает наши providers, значит слот открыт для сторонних data source и дополнительный "самсунговский" формат не нужен;
- если picker снова ограничен только Samsung/system sources, это нужно трактовать как private-ограничение конкретного face, а не как проблему нашего provider'а.

### Если используется собственный watch face

Тогда можно спроектировать слот под наши данные самостоятельно:

- создать arc/edge complication slot
- поддержать `RANGED_VALUE` / `GOAL_PROGRESS`
- протестировать отображение provider'а в picker'е

### Если используется stock Samsung watch face

Тогда поведение определяется самим Samsung watch face.  
Если системный picker не даёт выбрать сторонний источник для этого специального места, значит:

- либо слот не поддерживает наши типы
- либо слот зарезервирован логикой циферблата
- либо слот вообще не является пользовательским complication slot

В этом случае обойти ограничение со стороны provider'а нельзя.

## Решение / план действий

### Этап 1. Аудит текущего provider'а

Проверить:

- какие `ComplicationType` реально публикуются сейчас
- есть ли `RANGED_VALUE`
- есть ли `getPreviewData(...)` для всех заявленных типов

### Этап 2. Добавить поддержку gauge-like типов

Если сейчас поддерживаются только текстовые complications, добавить:

- `RANGED_VALUE`
- при необходимости `GOAL_PROGRESS`

### Этап 3. Протестировать на нескольких слотах

Проверить отдельно:

- обычные круглые/текстовые слоты
- специальные верхние/дуговые слоты
- один и тот же provider на разных циферблатах

Цель — понять, проблема в provider или в конкретном stock watch face.

### Этап 4. Зафиксировать ограничение платформы

Если после добавления подходящих типов provider всё равно не появляется только в специальных слотовых зонах stock Samsung face, зафиксировать это как ограничение конкретного watch face:

- provider реализован корректно
- обычные пользовательские слоты работают
- специальные слоты недоступны для сторонних data sources

## Итог

Для специальных слотов Samsung Watch Ultra-like циферблатов не нужен отдельный proprietary-формат Samsung.  
Нужно:

1. публиковать правильные `ComplicationType`
2. корректно реализовать provider и manifest
3. учитывать, что часть специальных слотов stock watch face может быть недоступна для сторонних providers вообще

Отдельно: обычный большой нижний слот, который работает через стандартный picker и принимает `LONG_TEXT`, можно считать нормальным рабочим сценарием для наших providers, а не отдельным Samsung-specific кейсом.

## Короткая формулировка для backlog

Исследовать совместимость custom complication provider со специальными слотами Samsung Ultra-style watch faces. Проверить поддержку `RANGED_VALUE` / `GOAL_PROGRESS`, корректность manifest и preview data. Если provider не появляется только в специальных слотах stock face, зафиксировать ограничение циферблата как внешнее платформенное ограничение.
