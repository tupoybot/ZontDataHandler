# Промпт для Codex на этап 7

```text
Ты работаешь в уже существующем Android-проекте, где этапы 1, 2, 3, 4, 5 и 6 уже завершены: рабочая вертикаль `phone -> watch -> complications` реализована, Samsung-specific и release-readiness правки предыдущих этапов уже внесены, а текущая кодовая база уже умеет локально собирать signed release при наличии keystore вне git.

Перед началом реализации обязательно прочитай:
- `docs/stage-07-plan.md`
- `docs/development-plan.md`
- `README.md`
- `docs/stage-06-plan.md`
- `.github/workflows/build-apks.yml`
- `wear`-код вокруг complication providers, formatter и preview data
- текущий код release signing / Gradle конфигурации

Сначала согласуй свою реализацию с этими документами, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких keystore, base64 secrets, паролей и токенов в git;
- приложение не нужно публиковать в Play Market;
- этапы 1-6 уже работают, их нельзя ломать;
- `wear`-приложение остаётся обязательной частью решения;
- этап 7 не про новые данные или новые пользовательские сценарии, а про bounded cosmetic polish плюс финализацию signed release path в CI;
- signed release через GitHub Actions должен стать practically achievable через secrets, а не оставаться только в теории.

Что уже должно остаться рабочим:
- синхронизация `phone -> watch`;
- текущие providers:
  - `ZONT overview`
  - `ZONT overview + icons`
  - `ZONT room`
  - `ZONT burner`
  - `ZONT setpoint`
  - `ZONT coolant`
  - `ZONT setpoint + coolant`
  - `ZONT room + air setpoint`
- текущая логика placeholder/stale;
- подтверждённый маппинг реального устройства:
  - `roomTemperature` — комнатная температура;
  - `burnerModulation` — модуляция горелки;
  - `targetTemperature` для подтверждённого `z3k-state` устройства — уставка теплоносителя (`setpoint_temp`), а не температура воздуха;
  - `coolantTemperature` — фактическая температура теплоносителя;
  - `roomSetpointTemperature` — отдельная желаемая температура воздуха в помещении.

Главный фокус этапа 7:
1. Провести bounded review текущего `LONG_TEXT` render path.
2. Слегка изменить визуальный вид "длинного" complication:
   - убрать стартовую пиктограмму в начале;
   - добавить пиктограмму значения перед каждым значением;
   - добиться того, чтобы пиктограмма стала визуальным разделителем.
3. Отдельно проверить typography `ZONT burner` и, если причина действительно в шрифте, выровнять размер шрифта показателя горелки с температурами воздуха и теплоносителя.
4. Подготовить и довести до практического состояния signed release через GitHub Actions.

Что нужно сделать:
1. Сначала проверь, какие именно providers и какие API-ветки сейчас формируют нужный `LONG_TEXT` complication.
2. Убедись, что запрошенный visual result вообще поддерживается актуальным Wear OS complications API проекта.
3. Если точная inline-пиктограмма внутри `LONG_TEXT` реально возможна, реализуй её минимально и аккуратно.
4. Если точная реализация ограничена платформой:
   - не маскируй это;
   - выбери ближайший bounded visual вариант;
   - объясни ограничение конкретно.
5. Отдельно проверь проблему размера шрифта у `ZONT burner`:
   - сначала убедись, что визуальная разница действительно связана с размером шрифта;
   - проверь, не создаётся ли эффект конкретным watch face, типом slot или расположением виджета;
   - только после этого вноси минимальную typography-правку.
6. Не ломай существующие providers, placeholder/stale и маппинг метрик.
7. Доведи GitHub Actions release path до понятного состояния:
   - workflow должен уметь восстанавливать keystore из secrets;
   - список secrets должен быть буквальным и недвусмысленным;
   - без secrets workflow должен честно оставаться unsigned.
8. Если правки затрагивают README, workflow, Gradle, release signing docs или stage 7 документы, обнови их согласованно.
9. После каждой значимой правки собирай и проверяй проект:
   - тесты;
   - `mobile`;
   - `wear`;
   - release path локально;
   - signed/unsigned distinction настолько, насколько это реально возможно в текущем окружении.
10. Если для полной проверки signed release через GitHub Actions нужны действия пользователя в GitHub Settings, остановись в нужный момент и запроси их явно, а не продолжай делать вид, что всё уже проверено.
11. Не делай commit, tag, merge или push без явной команды пользователя.

Ограничения:
- не добавляй новые пользовательские фичи;
- не переписывай архитектуру complications;
- не меняй смысл существующих метрик;
- не храни secrets в git;
- не обещай platform behavior, который не подтверждён кодом или проверкой;
- не превращай косметический этап в большой redesign.

Технические предпочтения:
- Kotlin;
- текущая архитектура проекта должна остаться узнаваемой;
- минимальные и понятные изменения без лишнего усложнения;
- review-first, затем bounded implementation;
- если нужен helper script для локальной подготовки signing materials, он может существовать, но outputs должны оставаться вне git.

Что важно по UX и продукту:
- пользовательский путь не должен стать хуже;
- complication должен стать аккуратнее, а не "более умным, но менее читаемым";
- `ZONT burner` не должен визуально выпадать из общей typographic иерархии, если проблема действительно в font size;
- watch-side поведение не должно регрессировать;
- signed release path должен выглядеть как реальный рабочий путь, а не как список общих пожеланий;
- если для включения signed CI нужны ручные действия в GitHub, они должны быть описаны буквально: куда зайти и какие secret names создать.

Что проверить перед завершением:
- проект собирается;
- тесты проходят;
- старая рабочая вертикаль не сломана;
- `LONG_TEXT` visual change либо реализован, либо platform limitation описан честно;
- отличие `ZONT burner` либо подтверждено как typography issue и исправлено, либо честно объяснено как следствие watch face / slot-specific layout;
- локальный signed release path работает;
- GitHub Actions подготовлен к signed release через secrets;
- unsigned fallback без secrets остаётся явно отмеченным;
- документация соответствует реальному состоянию проекта;
- commit/push не делался без команды пользователя.

Если часть запрошенного visual behavior окажется спорной из-за ограничений Wear OS complications API, выбирай самое простое решение, которое:
- не ломает существующую систему;
- делает complication понятнее;
- оставляет после себя честный технический результат;
- не скрывает platform limitation расплывчатыми формулировками.
```
