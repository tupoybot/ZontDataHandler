# Этап 2: план реализации для Codex

## Цель этапа

Сохранить уже работающую вертикаль `phone -> watch complications`, но сделать watch-side отображение заметно более нативным и пригодным для реального использования на Galaxy Watch:

- убрать визуально "технические" подписи;
- сделать combined complication компактным и читаемым в большом верхнем слоте;
- сделать отдельные complications естественными для круглых слотов;
- не сломать рабочую синхронизацию и корректный маппинг метрик.

## Что уже считаем завершённым

На этапе 1 уже подтверждено:

- проект собирается;
- телефон получает данные из ZONT;
- snapshot передаётся на часы;
- на часах доступны 5 complication providers;
- ручная проверка на эмуляторах прошла успешно;
- для подтверждённого `z3k-state`-формата:
  - `roomTemperature` берётся из датчика помещения;
  - `burnerModulation` берётся из `ot.rml`;
  - `targetTemperature` для реального устройства должно соответствовать уставочной температуре теплоносителя, то есть `setpoint_temp`, а не целевой температуре помещения;
  - `coolantTemperature` берётся из `ot.bt`.

Этап 2 не должен ломать этот уже подтверждённый путь.

## Известные баги, которые нужно учесть на этапе 2

- если в телефоне поставить интервал автообновления меньше `5` минут, например `1` минуту для теста, то после `Refresh` значение откатывается обратно на `5`;
- это нужно считать подтверждённым багом текущей реализации, а не ожидаемым поведением;
- на этапе 2 нужно либо исправить сохранение/нормализацию интервала, либо явно разделить:
  - продуктовый минимум по умолчанию;
  - тестовый минимально допустимый интервал для ручной проверки.

## Принятые ограничения

Чтобы не раздувать объём этапа 2:

- не переделываем архитектуру с нуля;
- не добавляем сервер, БД, логин/пароль и другую новую инфраструктуру;
- не делаем полноценный watch UI;
- не добавляем графики, историю и управление котлом;
- не пытаемся сделать универсальный дизайнер watch faces;
- не делаем сложную систему темизации.

## Главный UX-принцип этапа 2

Подписи в complications нужно убрать.

Практическое правило:

- в отдельных complications не показывать слова вроде `ROOM`, `TARGET`, `BURNER`, `COOLANT`;
- в combined complication не показывать строку вида `Room 23.7° | Burner 48% | ...`;
- максимум допустимы:
  - значение само по себе;
  - одна короткая пиктограмма;
  - одна максимально ненавязчивая общая метка, если без неё API или watch face выглядят хуже.

При конфликте между "объясняющей подписью" и "нативным видом" выбирать нативный вид.

## Scope этапа

### 1. Переработать presentation logic для complications

Нужно изменить watch-side форматирование так, чтобы:

- combined complication отдавал компактное представление без названий метрик;
- отдельные complications показывали только значение или значение + пиктограмму;
- stale и missing состояния оставались понятными, но тоже без лишнего текста.

### 2. Добавить минимальный набор пиктограмм

Если это упростит восприятие и действительно улучшит вид:

- добавить монохромные векторные иконки для:
  - room;
  - burner;
  - target;
  - coolant;
  - optional: общий combined icon.

Требование:

- иконки должны быть простыми и "watch-native", должны уметь подстраиваться под цветовую гамму дисплея;
- не использовать пёстрые изображения;
- не перегружать complication декоративными элементами.
- как пример возьми значения текущего пульса, погоды, батарейки

### 3. Адаптировать combined provider под большой слот Samsung watch face

Нужно добиться того, чтобы combined complication:

- хорошо выглядел в большом верхнем слоте Samsung watch face;
- по возможности вмещал 4 значения без визуального шума;
- корректно деградировал в `SHORT_TEXT`, если текущий face не даёт полноценный `LONG_TEXT`.
- на текущий момент galaxy watch ultra вообще не предлагают выбрать ZONT-complication на месте большого дополнения экрана ultra - надо изучить каким образом вообще оно выбирает подходящие, возможно там место только для samsung разработок, или нужно соблюдать специальный формат

Предпочтение для combined presentation:

- сначала пробовать формат только из значений, например в духе:
  - `23.7°  48%  46.8°  45.8°`
  - или `23.7° · 48% · 46.8° · 45.8°`
- если API/слот требует дополнительного элемента, лучше использовать одну иконку или краткий заголовок уровня `ZONT`, но не названия каждой метрики.

### 4. Адаптировать одиночные providers под круглые слоты

Одиночные providers должны быть пригодны для небольших круглых слотов:

- primary content — само значение;
- подписи в тексте убрать;
- title использовать только если без него конкретный тип complication смотрится хуже;
- при наличии иконки она не должна вытеснять значение.

### 5. Согласовать stale / missing presentation

Нужно оставить понятную деградацию данных, но без "технического" текста:

- отсутствующее значение: `--`;
- stale: короткий и ненавязчивый признак, например префикс, иконка или другой лаконичный вариант;
- не использовать длинные слова вроде `stale`, `missing`, `unavailable` в самих complications, если без них можно обойтись.

### 6. Немного подчистить watch debug/status screen

Экран статуса на часах можно оставить минимальным, но стоит:

- привести его форматирование в соответствие с новым complication style;
- убрать лишние длинные подписи, если они мешают;
- оставить его debug-friendly.

### 7. Документировать реальный способ тестирования

В README или в отдельном описании проверки нужно явно зафиксировать:

- как тестировать быстро на обычном Wear OS emulator;
- что для Samsung-специфичного визуального результата нужен реальный Galaxy Watch или Samsung Remote Test Lab;
- что Samsung Remote Test Lab для отладки через IDE может требовать отдельный шаг с `Remote Debug Bridge`;
- какие watch faces и какие слоты были проверены.

## Критерии приёмки этапа

Этап 2 считается завершённым, если:

- проект по-прежнему собирается;
- `phone -> watch` синхронизация по-прежнему работает;
- combined complication больше не показывает длинные подписи метрик;
- одиночные complications больше не показывают слова `ROOM`, `TARGET`, `BURNER`, `COOLANT` как основной текст;
- combined complication выглядит заметно чище в большом верхнем слоте;
- одиночные complications выглядят естественнее в круглых слотах;
- `targetTemperature` не регрессирует и для подтверждённого `z3k-state` устройства продолжает соответствовать уставке теплоносителя;
- missing/stale состояния остаются понятными;
- баг с откатом интервала автообновления после `Refresh` исправлен или явно переопределён и документирован;
- путь ручной проверки обновлён и понятен.

## Как тестировать на этапе 2

Минимальный путь:

- Wear OS emulator для быстрой итерации;
- Android emulator или реальный телефон для phone-side части;
- обязательная финальная визуальная проверка на `Galaxy Watch Ultra`, если он доступен.

Если реальных часов нет под рукой постоянно, разумный fallback:

- использовать Samsung Remote Test Lab для проверки на реальном Galaxy Watch.
