# Промпт для Codex на этап 6

```text
Ты работаешь в уже существующем Android-проекте, где этапы 1, 2, 3, 4 и 5 уже завершены: рабочая вертикаль `phone -> watch -> complications` реализована, визуальные и Samsung-specific правки предыдущих этапов уже внесены, а текущая базовая версия проекта собрана вокруг релиза `0.2`.

Перед началом реализации обязательно прочитай:
- `docs/stage-06-plan.md`
- `docs/development-plan.md`
- `README.md`
- `docs/stage-05-plan.md`
- `mobile/build.gradle.kts`
- `wear/build.gradle.kts`
- `.github/workflows/build-apks.yml`
- код вокруг `WorkManager`, `repository`, `Data Layer` и текущей release/debug сборки

Сначала согласуй свою реализацию с этими документами, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких секретов, keystore и паролей в git;
- приложение не нужно публиковать в Play Market;
- этапы 1-5 уже работают, их нельзя ломать;
- `wear`-приложение остаётся обязательной частью решения;
- этап 6 посвящён техническому ревью и рефакторингу, а не новым пользовательским функциям;
- текущая рабочая вертикаль должна стать надёжнее, понятнее и ближе к реальному release-ready состоянию;
- пользователь должен видеть установленный version/build на телефоне и часах, чтобы sideload/install path можно было проверять без догадок;
- если в ходе review обнаружатся скрытые технические проблемы вне исходного списка, их тоже нужно оценить и при возможности закрыть.

Что уже должно остаться рабочим:
- синхронизация `phone -> watch`;
- текущие providers:
  - `ZONT overview`
  - `ZONT overview + icons`
  - `ZONT room`
  - `ZONT burner`
  - `ZONT setpoint`
  - `ZONT coolant`
  - `ZONT setpoint + coolant`
  - `ZONT room + air setpoint`
- текущая логика placeholder/stale;
- подтверждённый маппинг реального устройства:
  - `roomTemperature` — комнатная температура;
  - `burnerModulation` — модуляция горелки;
  - `targetTemperature` для подтверждённого `z3k-state` устройства — уставка теплоносителя (`setpoint_temp`), а не температура воздуха;
  - `coolantTemperature` — фактическая температура теплоносителя;
  - `roomSetpointTemperature` — отдельная желаемая температура воздуха в помещении.

Главный фокус этапа 6:
1. Провести review-first аудит кода и найти реальные технические риски.
2. Усилить тестирование и проверить производительность там, где это действительно важно.
3. Сделать фоновую работу приложения более надёжной и предсказуемой в рамках ограничений Android.
4. Убрать технический мусор:
   - бесконечные логи;
   - шумные retry/reschedule-сценарии;
   - misleading debug-only поведение.
5. Привести release build path к понятному состоянию:
   - объяснить и оформить разницу между debug и release;
   - определить стратегию подписи release APK;
   - не хранить signing secrets в git;
   - не выдавать unsigned release за "готовый релиз".
6. Найти и по возможности закрыть другие скрытые архитектурные и технические проблемы, которые не видны на первый взгляд.

Что нужно сделать:
1. Начни с ревью ключевых потоков:
   - `settings -> refresh -> snapshot -> data layer -> complications`
   - `WorkManager` / scheduling
   - release/debug build pipeline
   - workflow / README / документация
2. Сформируй реальный список findings по приоритету и исправляй в первую очередь то, что влияет на:
   - надёжность;
   - battery/performance;
   - фоновую работу;
   - release readiness;
   - сопровождаемость.
3. Усиль тесты только там, где это действительно снижает риск регрессии.
4. Если найдёшь performance issue, сначала докажи его анализом или измерением, а потом оптимизируй.
5. Проверь текущую background-логику и исправь её так, чтобы она была максимально надёжной в рамках Android:
   - без магических обещаний "вечной" фоновой работы;
   - без дублирующихся работ;
   - без бесконечной шумной активности.
6. Проверь, что ошибки не маскируются тихим успехом там, где нужен retry, bounded status или явный отказ.
7. Отдельно проверь release path:
   - сейчас debug и release нельзя считать одинаково готовыми;
   - debug обычно подписывается стандартным debug keystore автоматически;
   - release для нормальной установки и обновлений должен быть подписан стабильным приватным keystore;
   - keystore должен жить вне git, например в local `keystore.properties` или через env vars;
   - если ключа нет, unsigned release допустим только как честно задокументированный промежуточный результат.
8. Не ломай отображение установленного version/build на телефоне и часах: оно нужно для ручной проверки sideload и install/upgrade-сценариев.
9. Если правки затрагивают README, workflow, Gradle-конфигурацию или документацию, обнови их согласованно.
10. После каждой значимой правки собирай и проверяй проект:
   - тесты;
   - `mobile`;
   - `wear`;
   - release path настолько, насколько это реально возможно в текущем окружении.
11. Не делай commit, tag, merge или push без явной команды пользователя.

Ограничения:
- не добавляй новые пользовательские фичи без необходимости;
- не переписывай архитектуру с нуля;
- не тащи сервер, аналитику, crash-reporting платформу или тяжёлую observability-инфраструктуру;
- не храни секреты, токены, keystore и пароли в git;
- не обещай фоновые гарантии, которые противоречат платформе;
- не добавляй логирование "на всякий случай", если оно создаёт мусор;
- не маскируй технические долги общими словами без конкретного результата.

Технические предпочтения:
- Kotlin;
- текущая архитектура проекта должна остаться узнаваемой;
- минимальные и понятные refactoring changes без лишнего усложнения;
- review-first, затем targeted fixes;
- при необходимости можно немного расширить тестовую инфраструктуру, но только bounded и по делу.

Что важно по UX и продукту:
- пользовательский путь не должен стать хуже;
- complications и watch-side поведение не должны регрессировать;
- ручной refresh и базовый sideload-сценарий должны остаться понятными;
- установленный version/build должен оставаться видимым на телефоне и часах;
- debug/release различия должны быть честно объяснены;
- release-путь не должен выглядеть как "что-то там когда-нибудь подпишем".

Что проверить перед завершением:
- проект собирается;
- тесты проходят;
- старая рабочая вертикаль не сломана;
- фоновые сценарии стали более предсказуемыми;
- нет бессмысленного log spam или бесконечной фоновой активности;
- release/debug различия и signing path описаны понятно;
- release APK либо действительно подписывается по понятной схеме, либо unsigned статус явно зафиксирован;
- документация соответствует реальному состоянию проекта;
- commit/push не делался без команды пользователя.

Если часть потенциальных технических проблем окажется спорной, выбирай самое простое решение, которое:
- снижает реальный риск;
- не ломает существующую систему;
- делает проект понятнее для следующего разработчика;
- оставляет после себя честный технический результат, а не декоративный "cleanup".
```
