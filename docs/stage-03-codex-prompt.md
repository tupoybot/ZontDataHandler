# Промпт для Codex на этап 3

```text
Ты работаешь в уже существующем Android-проекте, где этапы 1 и 2 уже завершены: рабочая вертикаль `phone -> watch -> complications` реализована, а текущие complication providers уже приведены к более нативному виду.

Перед началом реализации обязательно прочитай:
- `docs/stage-03-plan.md`
- `docs/development-plan.md`
- `docs/galaxy-specific.md`
- при необходимости текущую реализацию этапов 1 и 2

Сначала согласуй свою реализацию с этими документами, затем переходи к коду.

Контекст:
- репозиторий публичный, поэтому никаких секретов в git;
- приложение не нужно публиковать в Play Market;
- этапы 1 и 2 уже работают, их нельзя ломать;
- пользователь тестирует на Android emulator, Wear OS emulator, Galaxy S23+ и Galaxy Watch Ultra;
- `wear`-приложение остаётся обязательной частью решения;
- на этапе 3 главный фокус: удобство запуска, подготовка релиза и документация;
- Samsung-specific special slot — это вторичный bounded research, а не право сломать весь этап.

Что уже должно остаться рабочим:
- синхронизация `phone -> watch`;
- текущие 7 providers:
  - `ZONT overview`
  - `ZONT room`
  - `ZONT burner`
  - `ZONT setpoint`
  - `ZONT coolant`
  - `ZONT setpoint + coolant`
  - `ZONT room + air setpoint`
- текущая логика placeholder/stale;
- подтверждённый маппинг реального устройства:
  - `roomTemperature` — комнатная температура;
  - `burnerModulation` — модуляция горелки;
  - `targetTemperature` для подтверждённого `z3k-state` устройства — уставка теплоносителя (`setpoint_temp`), а не температура воздуха;
  - `coolantTemperature` — фактическая температура теплоносителя;
  - `roomSetpointTemperature` — отдельная желаемая температура воздуха в помещении.

Что нужно сделать:
1. Сохрани текущую рабочую вертикаль и не ломай существующие 7 providers.
2. Добавь в `mobile` более удобный способ получения рабочего токена ZONT по логину и паролю, если это можно реализовать простым клиентским HTTP-flow без новой инфраструктуры.
3. Сохрани ручной ввод токена как fallback, если auth-flow окажется нестабильным, частично undocumented или потребует нежелательных обходов.
4. Не храни логины, пароли, токены и другие секреты в git, `BuildConfig`, ресурсах или CI secrets без явной необходимости.
5. Добавь минимальный GitHub Actions workflow для сборки APK:
   - `mobile` и `wear`;
   - как минимум `debug`;
   - по возможности и `release`, если это можно сделать без тяжёлой signing-инфраструктуры;
   - артефакты должны публиковаться в GitHub Actions.
6. Приведи `README.md` в более чистый и короткий вид:
   - убери устаревшее;
   - оставь актуальные шаги сборки, установки и ручной проверки;
   - опиши текущие providers и реальный путь тестирования на emulator / Galaxy Watch;
   - добавь честную пометку, что кодовая база полностью сгенерирована Codex.
7. Подготовь релиз `0.1`:
   - проверь версию;
   - подготовь release notes или краткий changelog;
   - сделай чистый commit;
   - если из текущей среды реально доступно, создай tag/release `0.1`;
   - если публикация релиза наружу недоступна, подготовь всё для ручного выпуска и явно это зафиксируй.
8. Используй `docs/galaxy-specific.md` для ограничённой проверки special Samsung slot compatibility:
   - сначала сделай аудит текущих `ComplicationType`;
   - затем только самый простой и безопасный эксперимент, если он не ломает текущие providers;
   - не уходи в большую Samsung-only ветку.
9. Если Samsung special slot требует заметной отдельной разработки, watch-face-specific логики, обхода stock-face ограничений или крупного редизайна providers, остановись и оформи это как этап 4, не размывая этап 3.
10. Если в рамках Samsung-эксперимента добавляешь `RANGED_VALUE` или `GOAL_PROGRESS`, не ломай текущие текстовые сценарии и обязательно обнови `getPreviewData(...)`, manifest и ручные шаги проверки.

Ограничения:
- не переделывай архитектуру заново;
- не добавляй сервер, базу данных и лишнюю инфраструктуру;
- не делай web-scraping, browser-automation и другие тяжёлые обходы ради логина;
- не ломай текущие complication layouts ради Samsung-эксперимента;
- не используй реальные секреты в репозитории;
- не превращай этап 3 в бесконечное исследование одного watch face.

Технические предпочтения:
- Kotlin;
- текущая архитектура проекта должна остаться узнаваемой;
- использовать существующий Wear OS complications API проекта;
- минимальные и понятные изменения без переусложнения;
- CI должен быть простым и поддерживаемым.

Что важно по UX:
- текущий рабочий пользовательский путь не должен стать хуже;
- ручной token-flow должен остаться как fallback, если автоматизация логина не сработает;
- README должен стать короче и понятнее;
- релиз `0.1` должен быть оформлен как аккуратный внутренний milestone, а не как "ещё один случайный commit";
- если Samsung-specific результат отрицательный, это тоже полезный результат, но он должен быть ясно задокументирован.

Что проверить перед завершением:
- проект собирается;
- старая работающая вертикаль не сломана;
- данные по-прежнему доходят до часов;
- существующие 7 providers работают;
- подтверждённый маппинг метрик не регрессировал;
- auth-path либо реально стал удобнее, либо ограничение честно описано, а fallback сохранён;
- GitHub Actions workflow собирает APK artifacts;
- README обновлён;
- release `0.1` подготовлен настолько, насколько это возможно из текущей среды;
- Samsung-ветка либо дала небольшой безопасный результат, либо явно вынесена в этап 4.

Если какие-то детали auth-flow, CI release-signing или Samsung-specific complications спорные, выбирай самое простое решение, которое даёт практическую пользу и не тащит за собой новую инфраструктуру.
```
